

# 产品需求文档 (PRD)：反脆弱结构洞套利与增信引擎 v2.0

## 1. 产品概述

本系统是一个基于图计算与大语言模型（LLM）的自动化信息套利与工程增信平台。系统通过持续监听、抓取不同网络域（如开源社区与商业市场）的信息，利用图算法识别具有高套利价值的“结构洞”，并将其进行自动化重构、合规过滤与测试封装，最终输出带有服务等级协议（SLA）保证的工业级产品或高确定性情报，赚取风险转移溢价。

## 2. 核心技术栈与全局约束

|模块|技术栈与约束|核心设计考量|
|---|---|---|
|**并发与网络**|单进程 `asyncio` + `aiohttp`|轻量级、高并发，避免多进程上下文切换开销。|
|**限流控制**|全局令牌桶 (Token Bucket) 算法|控制 API 请求频率，防封禁，确保爬虫与请求的稳定性。|
|**图计算引擎**|纯 Python `NetworkX`|内存级拓扑运算，灵活轻量，无需外部图数据库依赖。|
|**时效管理**|节点/边 TTL (Time-To-Live) 机制|实时计算结构洞衰变率，淘汰过期拓扑连接。|
|**安全与测试**|原生运行环境 (无需物理隔离)|降低架构复杂度，通过代码静态扫描与标准 `pytest` 保障质量。|

## 3. 功能模块需求 (5-Tier Architecture)

### 3.1 数据摄取与合规层 (Sensory & Compliance Layer)

- **全局限流调度 (Rate Limiter)：**
    
    - 实现基于协程的全局令牌桶机制。所有 `aiohttp` 请求必须通过此桶获取 Token，支持动态调整速率（如 50 req/sec）。
        
- **数据清洗与格式化 (Data Formatting)：**
    
    - 通过正则表达式与基础 NLP 脚本对抓取的非结构化文本、代码片段进行预处理、去噪和格式统一化。
        
- **许可识别与幻觉绕过 (License Bypass & Hallucination Override)：**
    
    - 利用 LLM 或静态规则扫描项目许可协议（MIT, Apache 2.0 等）。
        
    - **幻觉绕过机制：** 鉴于 LLM 存在误判（幻觉）的可能性（例如将带有免责声明的 MIT 误判为严格限制协议），引入“置信度阈值”与“强制绕过”策略。若 LLM 判定失败但正则关键字（如 `Permission is hereby granted...`）匹配通过，允许系统自动绕过 LLM 的悲观判定，执行白名单放行。
        

### 3.2 图谱分析与衰变层 (Graph Computing Layer)

- **动态拓扑图构建：**
    
    - 使用 `NetworkX.DiGraph` 构建 A 域与 B 域的实体/信息流向图。
        
- **TTL 与边衰变计算 (Edge Decay)：**
    
    - 为每条边注入 `timestamp` 属性。系统设定 TTL（如 72 小时），后台任务定期修剪过期边（Pruning）。
        
    - 计算两节点间的最短路径变化率。若路径在 TTL 周期内呈指数级缩短，触发“收益递减熔断”，标记该结构洞已失效。
        
- **Burt 约束指数计算：**
    
    - 基于公式 Ci​=∑j​(pij​+∑q​piq​pqj​)2 量化节点的网络桥梁价值，定位核心套利节点。
        

### 3.3 认知与风控决策层 (Cognitive & Risk Decision Layer)

- **上下文提取与 LLM 评估：**
    
    - 将未失效的结构洞数据（如 A 域的新算法，B 域的痛点）组装为 Prompt 输入 LLM。
        
- **强制结构化输出：**
    
    - LLM 必须以严格的 JSON 格式输出决策报告，包含：`compliance_status` (合规状态)、`pitch_script` (B域推销话术)、`expected_margin` (预期利润率)、`max_drawdown` (最大回撤风险)、`execution_steps` (执行步骤)。
        

### 3.4 工程增信层 (Engineering Validation Layer)

- **自动化测试 (无需沙盒)：**
    
    - 针对 A 域抓取的代码或算法，系统自动生成测试用例（Unit/Integration Tests）。
        
    - 直接在当前单进程环境中利用 `subprocess` 或内部模块调用执行测试（由于信任开源平台的基础代码，且旨在降低架构复杂度，**取消物理隔离/Docker沙盒**）。
        
- **SLA 报告生成：**
    
    - 捕获测试覆盖率与异常处理结果，打包生成“工业级容灾与稳定性报告”，作为向 B 域索取“兜底溢价”的谈判筹码。
        

### 3.5 变现与熔断执行层 (Execution & Breaker Layer)

- **产品化输出：**
    
    - 将通过测试的代码或信息封装为 API 服务（SaaS 订阅）或深度报告（认知差变现）。
        
- **人类在环 (Human-in-the-Loop) 熔断器：**
    
    - 如果 JSON 报告中 `max_drawdown` 超过预设安全垫阈值，或涉及跨域自动化交易操作，系统强制挂起，生成 Webhook/飞书/钉钉工单，等待人类审核确认后再执行。

## 4. 实施阶段规划 (Phased Rollout)

- **Phase 1 (基础框架)：** 搭建单进程 `asyncio` + `aiohttp` 爬虫，实现全局令牌桶，完成基础的数据清洗与格式化。
    
- **Phase 2 (图与时效)：** 引入 `NetworkX`，构建带 TTL 修剪逻辑的动态图，实现 Burt 指数与衰变率计算。
    
- **Phase 3 (AI 与增信)：** 接入 LLM JSON 解析，实现“许可幻觉绕过”逻辑，并集成无需隔离的自动化测试流程。
    
- **Phase 4 (变现测试)：** 针对“AI算法界与下沉电商”场景进行端到端的模拟跑通。
