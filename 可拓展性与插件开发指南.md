# NetBroker 二次开发与可拓展性开发指南

NetBroker 拥有极其松耦合的 5 层事件管道架构。这非常适合以插件化的形式**堆叠和挂载无数个并行套利剧本（Playbooks）**。

若您希望在原有的系统上增加**新的数据源**（如定期爬取 GitHub Trending 榜单）或**新的变现端**（如将文案自动发送到钉钉群或小红书自动化），请遵循以下四大纵向步骤轻松完成扩充。

---

## Step 1: 在 Sensory 层开发新漏斗 (Crawler)

抓取端不限制协议形态，但所有挂起的轮询类爬虫**强烈建议必须继承** `AioCrawler`，以此获得该基类关联的 **全局令牌桶限流** 保护。如果短时间内请求并发过猛，桶会主动休眠你的协程避免把你的本机 IP 玩被动拉黑。

1. 在 `sensory/` 目录下新建如 `github_crawler.py`。
2. 在发起真实的 `aiohttp`/`httpx` 请求前，必须主动强制执行 `await self.rate_limiter.consume(1)` 划扣令牌。
3. 抓取到的冗余杂乱文本，利用 `sensory/cleaner.py` 里的 `extract_code_blocks` 无脑去脏处理。

```python
# 示例：sensory/github_crawler.py
from sensory.aio_crawler import AioCrawler

class GithubCrawler(AioCrawler):
    async def fetch_trending_readme(self):
        await self.rate_limiter.consume(1) # [绝对禁止遗漏]
        # 执行目标域抓取并返回组装后的 dict
        return data_dict
```

## Step 2: 在 Cognitive 层开发专用的大模型认知逻辑

每一段“脏内容”被抓回后，需要大模型作为最强编译器帮它降维或变换范式。
进入 `cognitive/llm_gateway.py`，针对您的新业务场景新增一个处理方法（例如 `evaluate_github_repo`）。

1. 我们目前的统一结构出口叫 `CognitiveDecisionOutput`（请在 `json_validator.py` 里按需补齐新字段并设定默认值），我们对 Gemini 使用 `response_mime_type="application/json"`，永远不需要担心大模型格式错位出 Bug。
2. 在 System Prompt 定制化诉求。举例：将代码转化为 TikTok 脚本：
   ```python
   async def evaluate_github_script(self, raw_readme: str) -> CognitiveDecisionOutput:
       # Prompt：你是一个金牌自媒体编导，请分析以下这个开源项目
       # 将其转化为适合拍摄30秒短视频的分镜和解说话术...
       # 执行 self.client.models.generate_content
       pass
   ```

---

## Step 3: 在 Execution 层开发新的输出变现端 (Sinks)

来到水管最后一步：`execution/` 文件夹。将清洗并验证好的产出进行实际对外发射或保存。

1. 如果是落盘（像 Obsidian）：建立新的 `my_file_writer.py`。 
2. 如果是发请求（像 Webhook 或发邮件）：建立 `feishu_bot.py`，暴露对外方法用于组装最终产物。
3. **安全熔断防守**：如果是高危场景（比如需要用 RPA 跨平台帮用户下单交易自动化），在执行动作前，一定要执行 `await risk_breaker...` 若存在潜在财务风险，一定要调用 `execution/human_in_loop.py`，通过异步机制 `await request_human_approval` 把当前这一个循环挂起，等管理员回复了真指令才能继续往下走。

---

## Step 4: 将所有部件在 `core/engine_loop.py` 进行插排与接通 (Wiring)

一切部件散件就绪后，打开总线电箱 `engine_loop.py` 把它们串联组网。

1. **接驳实例**：在 `NetBrokerEngine.__init__` 当中实例化：
   ```python
   self.gh_crawler = GithubCrawler(self.token_bucket)
   ```
2. **挂载发动机协程**：
   - 建立一个 `_github_polling_task(self)` 方法，通过内嵌强大的 `while self.is_running:` 循环不断去执行抓取。抓到数据后，封装它的身份属性：
     `state=ArbitrageState.DISCOVERED` 并全数放入中央蓄水池 `self.pending_holes` 中！
   - 在已有的 `start()` 下使用 `asyncio.gather` 把你新建的 `_github_polling_task` 丢进总事件池运转。
3. **在评价中心引流 (Router)**：
   - 统一在 `_evaluation_task` 里使用一个分流路由。根据 `hole` 的某个参数区分来源；如果是 GitHub 库抓来的事件，直接转入给：
     `await self.llm_gateway.evaluate_github_script()`，
   - 并在取得完备的模型回复对象后，推送进您写的 `tiktok_publisher.execute_upload()` 中...最后设置本循环洞熔断闭环 `ArbitrageState.MELTED`。
